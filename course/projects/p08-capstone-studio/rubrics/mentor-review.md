# Capstone Mentor Review Rubric

## Blocking conditions

- The primary product is a generic chatbot.
- Model output or untrusted external text can grant authority.
- Consequential actions lack durable out-of-band approval.
- GitHub or the second API is decorative rather than outcome-relevant.
- LangGraph, Redis, or FastMCP is included without a defended need.
- Evaluation begins after implementation or tunes against the holdout.
- Claims omit denominators, baselines, or important regressions.
- Default verification requires live paid services.
- Customer package imports repository-only files or ships secrets/state.
- “Production ready” is claimed without operating evidence and limitations.
- The learner cannot explain or modify selected generated code.

## Gate sequence

1. G9A proposal defense and changed-constraint exercise.
2. Walking-skeleton review.
3. Evidence and API boundary review.
4. Durability, approval, replay, and recovery review.
5. Evaluation baseline and two-iteration review.
6. Package, container, red-team, and readiness review.
7. G9C portfolio, private scenario, code explanation, and mock interview.

## Passing standard

The learner demonstrates mid-level ownership: they translate ambiguous needs
into contracts, constrain model authority, diagnose failures with evidence,
course-correct AI-generated code, operate across backend and integration
boundaries, and communicate exactly what the prototype does and does not prove.
