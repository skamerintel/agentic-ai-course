# M00: Course Operating Model and Baseline Diagnostic

## Learning objectives

By the end of this module, you can:

- Distinguish AI-generated output from engineering evidence.
- Inspect a repository before asking a coding agent to modify it.
- Explain an unfamiliar project's entry points, boundaries, checks, and risks.
- Maintain a concise AI work log.
- Describe how automated, AI, and mentor reviews differ.

## Prerequisite diagnostic

Without using a coding agent, inspect the root of this repository for ten
minutes. Write down:

1. What the repository is for.
2. Which files define the course scope.
3. Which file controls solution access.
4. What is executable today.
5. Three unanswered questions.

You are not graded on completeness. This is the baseline against which your
later repository analysis will be compared.

## Concept lesson

### Fluent is not verified

A coding agent can produce a convincing repository summary after inspecting
only a few filenames. Treat that summary as a hypothesis. Verify it against:

- Actual entry points.
- Imported dependencies.
- Tests and quality commands.
- Configuration files.
- Data flow and external calls.
- Git history or pull-request context when available.

The professional habit is not distrust for its own sake. It is matching the
strength of a claim to the strength of its evidence.

### Inspect before changing

Before authorizing edits, establish:

1. The requested outcome.
2. The files and components likely involved.
3. Existing conventions and checks.
4. Current uncommitted changes.
5. Risks and unknowns.

Ask the agent to report evidence such as filenames, symbols, and commands. A
summary without evidence is useful for orientation but insufficient for a
decision.

### The ownership test

You own generated work only when you can:

- Explain why it exists.
- Trace its important paths.
- State its assumptions.
- Demonstrate its behavior.
- Identify likely failures.
- Change it intentionally.

If you cannot do these things, the code may run, but it is not yet your work.

### AI work log

Use the [work-log template](../../templates/ai-work-log.md). Record decisions,
not every prompt. A useful entry captures:

- The task given to the agent.
- The important proposal or change.
- What you verified.
- What you rejected or corrected.
- What you learned.

## Guided lab: evidence-backed repository survey

Inspect the Project 1 starter area after it is introduced by the mentor. Ask
your coding agent for a repository survey with this contract:

```text
Inspect without editing. Identify entry points, configuration, tests, data,
external-service boundaries, incomplete areas, and commands that appear safe to
run. Cite a path for every claim. Separate facts from inferences and questions.
```

Verify at least five claims manually. Mark each as confirmed, corrected, or not
yet verifiable.

## Independent challenge

Create `artifacts/m00/repository-inventory.md` containing:

- Purpose and current implementation status.
- Important paths and their responsibilities.
- Executable commands and expected effects.
- Trust boundaries and external systems.
- Existing automated checks.
- Five risks or unanswered questions.
- A section titled “Agent claims I corrected.”

Do not ask the agent to write the final inventory. It may critique your draft.

## Failure-analysis exercise

Review [the inaccurate repository summary](inaccurate-repository-summary.md).
For every inaccurate or unsupported statement:

1. Quote or summarize the claim.
2. Classify it as false, unsupported, stale, or overly broad.
3. Cite repository evidence.
4. Write a corrected version.

## Comprehension gate

The learner must provide the inventory and work log, then answer without agent
assistance:

- What is implemented versus merely planned?
- Which files are authoritative when two documents disagree?
- What command would you run first, and why is it safe?
- What would you inspect before accepting an agent-authored change?
- Name one agent claim you corrected and the evidence used.

## Interview questions

1. How do you evaluate code generated by an AI coding agent?
2. What is the difference between a plausible explanation and engineering
   evidence?
3. How do repository-level instructions improve or harm agent performance?
4. What makes an AI-assisted pull request reviewable?

## Required GitHub evidence

- Repository inventory.
- AI work log.
- Corrected failure-analysis exercise.
- A pull request containing only M00 artifacts.
- Completed AI and mentor rubrics.
