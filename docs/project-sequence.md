# Project Sequence

Status: **Design baseline**  
Curriculum map: [Curriculum roadmap](curriculum-roadmap.md)

## 1. Project design rules

Every project uses a supplied realistic dataset, a simulated business owner,
and incomplete or imperfect requirements. The learner must clarify assumptions,
write acceptance criteria, and maintain a concise AI work log.

Each repository must contain:

- A business-facing problem statement.
- Explicit scope and non-goals.
- Setup and reproducibility instructions.
- Tests and quality commands.
- An architecture or workflow diagram.
- Ground truth or deterministic acceptance fixtures.
- An AI-review rubric.
- A mentor-review rubric.
- An AI work log.
- A failure-analysis document.
- A pull request used for final review.
- A short portfolio summary and draft résumé bullet.

Reference implementations are available according to `course.toml`.

## 3. P1: Model API Behavior Lab

### Business scenario

A software company is selecting a model interface for converting unstructured
customer incident reports into concise internal summaries. The engineering lead
wants a reproducible comparison rather than anecdotal model preferences.

### Supplied assets

- A representative set of incident reports with short human-authored reference
  summaries.
- A task rubric covering factual fidelity, omitted critical details, unsupported
  claims, format adherence, latency, and usage.
- A flawed AI-generated comparison report for the failure exercise.

### Required implementation

- Call the OpenAI Responses API directly.
- Implement the equivalent bounded task with Chat Completions for interface
  comparison.
- Call the Anthropic Messages API for the same task.
- Store raw run metadata separately from normalized experiment results.
- Run repeated trials where nondeterminism matters.
- Produce a comparison grounded in results and documented limitations.

### Coding-agent constraint

The coding agent may generate client boilerplate only after the learner writes
the experiment contract, measurement fields, and acceptance criteria.

### Intentionally difficult elements

- Some reference summaries are imperfect.
- Several examples contain irrelevant but attention-grabbing details.
- Provider responses cannot be normalized losslessly.
- A cheaper or faster result may be good enough for the narrow task.

### Gate

The learner must:

- Trace one request and response for each API.
- Explain the important API-shape differences.
- Identify at least two unsupported claims in the flawed comparison.
- Make a recommendation that acknowledges uncertainty.

### Portfolio value

Demonstrates direct API literacy and disciplined model evaluation without
claiming universal model superiority.

## 4. P2: Typed Intake Normalizer

### Business scenario

An operations team receives service requests through email exports, forms, and
copied chat messages. Downstream automation requires a trustworthy typed record,
but inputs are incomplete and inconsistent.

### Supplied assets

- A corpus of messy service requests.
- Business rules for urgency, requester identity, requested action, affected
  system, dates, and missing-information handling.
- Edge cases containing contradictory dates, ambiguous urgency, malicious-looking
  text, and unsupported details.
- A deliberately weak Pydantic schema generated by an AI.

### Required implementation

- Define Pydantic input, output, error, and audit models.
- Use OpenAI structured output.
- Distinguish model refusal, provider failure, schema validation failure, and
  business-rule failure.
- Support deterministic batch processing with safe retry behavior.
- Record usage, latency, and correlation identifiers with redacted logs.

### Coding-agent constraint

The learner must author or dictate the schema rationale and adversarial examples
before asking the agent to implement the pipeline.

### Intentionally difficult elements

- Optional data is not always harmless to omit.
- Coercion can turn invalid identifiers or dates into apparently valid values.
- Retrying cannot repair every failure category.
- The desired output contract changes once during the project.

### Gate

The learner must defend every field, validation rule, retry decision, and
logged value. The mentor provides one unseen input that the learner must trace
through the system.

### Portfolio value

Demonstrates reliable structured extraction, runtime validation, and
failure-aware model integration.

## 5. P3: Operations Tool Agent

### Business scenario

An internal operations analyst needs a system that can answer bounded questions
by consulting service status, asset inventory, and incident-history tools. The
system must not modify external state.

### Supplied assets

- Deterministic local APIs for service status, inventory, and incident history.
- A scenario set with expected tool choices and final-answer requirements.
- Tool failures, stale data, duplicate records, and empty-result cases.
- A broken agent implementation that can loop indefinitely.

### Required implementation

- Build the agent loop directly without LangGraph or another agent framework.
- Define narrow, typed, read-only tools.
- Validate all model-generated tool arguments.
- Correlate tool results to their calls.
- Enforce iteration and repeated-call limits.
- Separate tool errors from model errors.
- Produce a complete run trace suitable for debugging.
- Return “insufficient evidence” rather than inventing an answer.

### Coding-agent constraint

The coding agent may implement the loop after the learner submits a state and
transition diagram. The learner must then compare the implementation to the
diagram and reject mismatches.

### Intentionally difficult elements

- Two tools have superficially similar purposes.
- Some questions cannot be answered from available tools.
- One tool intermittently fails.
- One scenario tempts the model to repeatedly call the same tool.

### Gate

The learner walks through successful, recoverable-failure, terminal-failure,
and loop-limit traces and explains every transition.

### Portfolio value

Demonstrates understanding of agent mechanics beneath frameworks.

## 6. P4: Eval-Driven Issue Triage Engine

### Business scenario

A development organization wants assistance triaging incoming GitHub issues.
The first version recommends labels, urgency, owning team, duplicate candidates,
and missing-information questions. It does not write to GitHub.

### Supplied assets

- A sanitized corpus of realistic issues, repository metadata, labels, and team
  ownership rules.
- A seed set of human labels containing several disagreements and mistakes.
- A held-out regression set controlled by the mentor.
- Several issue clusters requiring context or retrieval.

### Required implementation

- Define the task and labeling policy before implementation changes.
- Create a versioned evaluation dataset with meaningful slices.
- Build a baseline and report multiple metrics rather than one score.
- Maintain a failure taxonomy.
- Complete at least two controlled improvement iterations.
- Compare full-context, structured lookup, and retrieval where applicable.
- Prevent evaluation examples from leaking into prompts as hidden answers.
- Produce recommendations with evidence and uncertainty.

### Coding-agent constraint

The learner may use the coding agent heavily for evaluation infrastructure, but
must personally approve labels, slices, metrics, and conclusions.

### Intentionally difficult elements

- Ground truth is not perfectly objective.
- Aggregate accuracy hides poor performance for one important team.
- Retrieval can return a semantically similar but irrelevant issue.
- One apparent improvement is caused by an accidental dataset change.

### Gate

The learner presents baseline failures, experiment history, before/after
results, limitations, and held-out performance. A result that improves only the
visible development set does not pass.

### Portfolio value

Demonstrates the evaluation and critical-thinking practices that distinguish an
agent engineer from a prompt-demo builder.

## 7. P5: Asynchronous Agent Job Service

### Business scenario

The issue-triage engine must become a backend service that accepts work,
processes it asynchronously, preserves an audit history, and exposes progress
and results to a lightweight client.

### Supplied assets

- API requirements and example client interactions.
- PostgreSQL and Redis development services.
- Load scenarios containing duplicate submissions, cancellation, restarts, and
  slow model calls.
- An AI-generated monolithic FastAPI implementation for refactoring.

### Required implementation

- Expose typed FastAPI endpoints for job submission, status, cancellation, and
  result retrieval.
- Keep route handlers separate from business and provider logic.
- Persist jobs, attempts, outcomes, and relevant audit data in PostgreSQL.
- Use Redis only for a documented transient concern such as idempotency,
  progress, or short-lived coordination.
- Support streamed or polled progress.
- Use correlation identifiers across API, model, and persistence logs.
- Include unit, integration, and smoke tests with no required live model call.
- Document the limitations of in-process background work and the path to a
  dedicated worker architecture.

### Coding-agent constraint

The coding agent may propose the architecture, but the learner must compare at
least two state-management designs before implementation is approved.

### Intentionally difficult elements

- Clients retry submissions after timeouts.
- The service can restart during a model call.
- Redis data may expire.
- The model provider may succeed after the client disconnects.
- Cancellation races with completion.

### Gate

The learner traces state across PostgreSQL, Redis, logs, and HTTP for duplicate,
cancelled, failed, and completed jobs. The learner must justify each persistent
and transient data choice.

### Portfolio value

Demonstrates backend engineering around model calls rather than only model API
usage.

## 8. P6: Durable Release Readiness Workflow

### Business scenario

A release manager needs a workflow that gathers release metadata, checks issue
and pull-request state, evaluates release notes, identifies blockers, and pauses
for human approval before producing a final readiness decision.

### Supplied assets

- Release manifests, mock repository data, policy rules, and approval scenarios.
- Tools with both deterministic and model-assisted checks.
- Crash and replay fixtures.
- A set of ambiguous blocker policies requiring clarification.

### Required implementation

- Use LangGraph as the orchestration framework.
- Define typed graph state and explicit deterministic/model-assisted nodes.
- Persist checkpoints and resume by workflow identity.
- Stream meaningful progress events.
- Interrupt before the final consequential decision.
- Allow approve, edit, and reject paths.
- Recover after a simulated crash.
- Avoid repeating completed side effects after resume.
- Compare the graph to the hand-built agent loop from P3.

### Coding-agent constraint

The learner must approve the state schema and graph diagram before allowing the
coding agent to implement nodes and edges.

### Intentionally difficult elements

- A tool result changes between initial execution and resume.
- A node is safe to retry while another is not.
- An AI-generated graph contains redundant model calls.
- An edited human decision invalidates one downstream assumption.

### Gate

The learner demonstrates normal completion, rejection, edited approval, crash
recovery, and duplicate-side-effect prevention.

### Portfolio value

Demonstrates professional LangGraph orchestration rather than a simple ReAct
demo.

## 9. P7: GitHub Workflow MCP Service

### Business scenario

An engineering organization wants a reusable MCP service that exposes a narrow
set of GitHub repository and pull-request capabilities to compatible agents.
Read operations may run directly; write operations must be proposed and
approved.

### Supplied assets

- A dedicated GitHub sandbox repository.
- GitHub REST and webhook fixtures.
- Required read capabilities and candidate write capabilities.
- Redelivery, rate-limit, invalid-signature, permission, and stale-data cases.

### Required implementation

- Build a FastMCP server with typed tools and bounded results.
- Build a deterministic FastMCP client test suite independent of a model.
- Read repository, issue, and pull-request information through the GitHub API.
- Receive and validate at least one GitHub webhook event.
- Deduplicate webhook deliveries.
- Implement at least one guarded write path with explicit human approval.
- Explain authentication and least-privilege requirements without implementing
  a complete enterprise auth system.
- Package customer-facing code with `uv` and Hatchling.
- Verify the wheel or source distribution in a clean environment.
- Run the service in Docker and provide a Compose-based local evaluation path
  where supporting services are needed.
- Include correlated logs, health checks, and an external smoke test.

### Coding-agent constraint

The coding agent may implement substantial portions, but the learner must review
the tool surface, GitHub permissions, package contents, Docker build context,
and every write boundary manually.

### Intentionally difficult elements

- The initial tool catalog is too broad.
- Webhooks can arrive more than once or out of order.
- A read response can contain untrusted text intended to influence an agent.
- The generated package initially includes fixtures and local configuration.
- The first Dockerfile runs development source rather than the installed wheel.

### Gate

The learner demonstrates deterministic MCP tests, a real client connection,
signed webhook handling, deduplication, guarded writes, clean package
installation, and a container smoke test. The learner also presents a
prototype-to-production review.

### Portfolio value

Demonstrates MCP, GitHub integration, API safety, packaging, Docker, testing, and
professional delivery in one bounded service.

## 10. Capstone

### Domain-selection rule

The learner selects the business domain during M25, after completing enough of
the course to make an informed choice. Project 8 supplies four candidate briefs,
a selection matrix, scenario seeds, and a custom-domain path. Support escalation
investigation is the recommended default. The selected project must solve a
workflow problem and cannot be primarily a chatbot.

### Required shape

The capstone must:

- Use OpenAI through a defined provider boundary.
- Use LangGraph for a justified stateful or multi-step workflow.
- Integrate meaningfully with GitHub and at least one additional external API.
- Use FastAPI, Pydantic, PostgreSQL, and Redis where justified.
- Include a FastMCP component when it improves interoperability; MCP must not be
  added merely to satisfy a checklist.
- Include ground truth, repeatable evaluation, failure slices, and at least two
  measured improvement iterations.
- Include human approval before consequential actions.
- Use useful, redacted, correlated logging.
- Produce a clean `uv` and Hatchling package.
- Run in Docker with a documented smoke test.
- Include unit, integration, contract, and recovery-oriented tests in
  proportion to the risks.
- Produce the portfolio artifacts defined in the course specification.

### Candidate domains

These remain prompts for learner discovery, not pre-solved applications:

- Repository maintenance and issue-to-implementation planning.
- Pull-request risk assessment and release coordination.
- Dependency-upgrade investigation and approval workflow.
- Support escalation investigation across tickets, repositories, and status
  APIs.
- Engineering incident evidence collection and follow-up tracking.
- Compliance evidence gathering from repositories and approved business APIs.

Project 8 implements proposal and portfolio evidence validators, milestone
contracts, review rubrics, a gated exemplar, and private mentor challenges. It
does not provide a finished capstone product implementation.

### Proposal gate

Before implementation, the learner must defend:

- Business user and measurable outcome.
- Why an LLM or agent is justified.
- Deterministic versus model-assisted steps.
- Ground truth and evaluation plan.
- Tool and API boundaries.
- Data retention and logging policy.
- Human-approval points.
- Failure and recovery strategy.
- Scope that can be completed to portfolio quality.

### Final gate

The capstone passes only after:

- Automated checks pass from a clean environment.
- The AI reviewer applies the project rubric.
- The learner addresses or explicitly rejects review findings with reasons.
- The mentor reviews the pull request and artifacts.
- The learner demonstrates a success path and at least two failure paths.
- The learner explains generated code selected by the mentor.
- The learner presents evaluation evidence and production gaps honestly.

## 11. Dataset and fixture backlog

Course implementation must create and version these reusable asset packs:

1. Incident-report and reference-summary corpus for P1.
2. Multi-channel service-intake corpus for P2.
3. Deterministic operations APIs and scenario set for P3.
4. Sanitized GitHub issue corpus, labels, teams, and held-out set for P4.
5. Job-lifecycle and concurrency scenarios for P5.
6. Release manifests, policies, and crash/replay scenarios for P6.
7. GitHub API responses, webhook payloads, and sandbox-repository setup for P7.
8. Shared malformed model responses, provider errors, rate-limit responses, and
   adversarial text fixtures used across projects.

Datasets should be small enough to inspect manually but varied enough to expose
meaningful failure modes. Any synthetic data must be labeled as synthetic and
reviewed for unrealistic shortcuts.
